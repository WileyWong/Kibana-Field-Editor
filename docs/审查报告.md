## 🟡 中等问题

### 2. manifest.json 中 `permissions` 可能多余

方案文档 3.1 的 manifest 声明了 `"permissions": ["scripting"]`。`scripting` 权限用于通过 `chrome.scripting` API **动态注入** content script。但该 manifest 已通过 `content_scripts` **静态声明**了 content.js，不需要此权限。建议移除或确认是否有其他用途。

### 3. `postMessage` 使用 `'*'` 作为 target origin

方案文档 3.2 中 `window.postMessage(..., '*')` 缺乏 origin 校验。虽然 Chrome 扩展 + injected 脚本场景下实际风险较低，但更安全的做法是指定具体 origin（如 `location.origin`），或在接收端校验 `event.origin`。

## 🟢 轻微问题 / 建议

### 4. 需求文档 FR-05 和 FR-06 存在微小歧义

- FR-05 说"嵌套对象内的标量字段"可编辑 → 正确
- FR-06 说"点击在对象或数组的 key 名上"不触发 → 正确
- 但未明确：**点击一个值是对象/数组的字段（非其子字段）时**的行为。方案文档 3.3.2 `resolveEditTarget` 已明确了规则（value 是对象 → 不触发），建议需求文档也补充此条。

### 5. `callProxy` 路径拼接缺少空值兜底

方案文档第 339 行：

```js
callProxy(`/${ctx._index}/_update/${encodeURIComponent(ctx._id)}`, 'POST', body)
```

若 `_index` 来自 ES hit 中的 `_index` 字段，正常场景没问题。但若 `_index` 为空或格式异常，缺少兜底校验。建议增加空值保护。

### 6. FR-21 与 FR-19 存在微妙状态问题

FR-19 说"将 Kibana 结果区…该字段的值**局部替换**为新值"，FR-21 说"弹窗关闭后，用户可继续点击其他字段进行修改"。但如果用户**替换后不关闭弹窗**而点击了结果区中的另一个字段——当前交互没有定义这个场景，可能导致两个浮层或弹窗状态冲突。
